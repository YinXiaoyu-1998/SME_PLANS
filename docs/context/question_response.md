# 企业资料中枢设计问答记录

本文档记录围绕“企业资料中枢”进行的设计追问、回答和决策。前半部分来自用户与主 agent 的共识；后续部分记录架构 subagent Curie 的判断。

## 已确认决策

### Q21：上传资料后，默认是否进入可检索状态？

**决策**：选择“处理完成后才活跃”。

**记录**：
- 资料上传后先进入待处理状态。
- 只有必要处理完成后，资料才成为活跃资料。
- 待处理资料不参与普通查询、搜索和员工智能体检索。

**理由**：
- 第一阶段更简单，不需要向普通员工解释“文件存在但内容索引还没好”的中间状态。
- 普通查询结果保持一致：能查到的资料就是已经准备好的资料。

## 架构 Subagent：Curie

任务：作为有 AI 智能体、知识库、数据库复杂系统设计经验的架构工程师，基于 `CONTEXT.md` 和 `architecture-designer` skill 继续推进企业资料中枢第一阶段系统设计。

结论：第一阶段建议采用 **模块化单体 + 异步处理 worker + MySQL 元数据主库 + 对象存储文件本体 + 可重建派生索引**。不要过早拆微服务，也不要把企业资料中枢做成智能体运行平台。

### Q22：上传后资料状态机如何设计？

**Curie 推荐**：`uploading -> pending_processing -> processing -> active | processing_failed -> archived`。只有 `active` 进入普通查询、搜索和向量检索；`processing_failed` 只对上传者和管理员可见。

**理由**：避免“文件已上传但索引未完成”被误用；状态简单，便于排障。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q23：上传成功是否等于资料创建成功？

**Curie 推荐**：不等于。上传成功只代表文件已进入对象存储并创建资料目录记录；业务可用必须等处理完成并转为 `active`。

**理由**：把存储成功和业务可检索成功分开，减少脏数据进入知识库。

**记录方向**：写入 `CONTEXT.md`；并入状态机 ADR。

### Q24：处理失败后是否自动重试？

**Curie 推荐**：自动重试 3 次，指数退避；仍失败则进入 `processing_failed`，保留错误码、错误摘要、最后失败时间和重试次数。管理员可手动重试。

**理由**：临时网络、OCR、解析服务失败常见；无限重试会制造运维噪音。

**记录方向**：写入 `CONTEXT.md`。

### Q25：处理流程是否必须幂等？

**Curie 推荐**：必须。每个资料记录有稳定 `document_id`，每次处理生成 `processing_run_id`；派生索引写入前按 `document_id + chunk_hash/index_type` 去重，或先清理本次可重建索引。

**理由**：重试、worker 崩溃、人工重跑都会发生；不幂等会造成重复向量、重复结构表和错误召回。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q26：第一阶段服务拆分怎么做？

**Curie 推荐**：一个后端应用内做模块化单体，模块包括账号权限、资料目录、上传处理、检索查询、审计、后台管理、Skill 目录。异步 worker 可作为同一代码库的独立进程。

**理由**：团队和业务规模不需要微服务；模块边界先清楚，部署保持简单。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q27：什么时候才需要拆服务？

**Curie 推荐**：只有当上传处理吞吐、向量检索、权限查询或后台管理出现明确独立扩展需求时再拆；第一候选拆分是上传处理 worker，其次是检索服务。

**理由**：按瓶颈拆，而不是按想象中的组织结构拆。

**记录方向**：作为技术路线记录，不必写入领域词汇。

### Q28：MySQL 的职责是什么？

**Curie 推荐**：MySQL 是权威元数据和权限判断来源，保存员工账号、标签、资料目录记录、状态机、文件对象位置、审计日志、处理任务、Skill 目录、经营事件元数据。

**理由**：这些数据需要事务、一致性、可追溯和权限过滤。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q29：对象存储的职责是什么？

**Curie 推荐**：只保存原始资料和分析产物文件本体，路径不可作为权限依据；权限永远通过 MySQL 资料目录记录判断。

**理由**：对象存储适合大文件，不适合表达业务权限；避免绕过目录记录直接访问。

**记录方向**：写入 `CONTEXT.md`；并入存储职责 ADR。

### Q30：向量表/向量库的职责是什么？

**Curie 推荐**：向量索引是派生索引，不是权威资料库。每条 chunk 必须带 `document_id`、可校验来源状态的字段或关联、权限过滤字段、`chunk_hash`、模型版本。向量命中后仍要做权限和状态校验。

**理由**：归档、权限变化、重建索引都需要追溯来源。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q31：结构化数据集放哪里？

**Curie 推荐**：第一阶段优先记录为 MySQL 中的受管数据表或外部表元数据；大体量明细可进入专门 schema，但必须由资料目录记录追溯来源。

**理由**：中小企业初期体量不大，MySQL 足够；过早引入数仓会增加运维成本。

**记录方向**：写入“结构化数据集必须可追溯来源”。

### Q32：查询入口应该有哪些？

**Curie 推荐**：第一阶段提供三类入口：HTTP API 作为稳定核心；MCP 作为员工智能体优先入口；CLI 作为管理员和调试入口。MCP/CLI 都调用同一套后端 API。

**理由**：API 保持系统边界清晰，MCP 服务智能体，CLI 降低早期后台复杂度。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q33：查询 API 应该返回文件还是检索结果？

**Curie 推荐**：分层提供目录查询、全文/向量检索、文件下载签名 URL、结构化数据查询。默认只返回当前员工有权访问且 `active` 的资料。

**理由**：员工智能体既需要“找资料”，也需要“拿文件”或“查结构化数据”。

**记录方向**：写入 `CONTEXT.md`。

### Q34：鉴权方式怎么做？

**Curie 推荐**：第一阶段使用员工账号登录后签发短期 JWT 或 session；MCP/CLI 使用员工个人 token 或 OAuth device flow，token 绑定员工账号，不创建独立智能体身份。

**理由**：符合“员工账号是唯一权限主体”的共识，同时方便工具调用。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q35：服务间/worker 鉴权怎么做？

**Curie 推荐**：内部 worker 使用服务凭证，只能处理任务和写派生索引，不能绕过资料状态机直接发布资料；所有最终可见性仍由资料目录状态决定。

**理由**：内部组件不应拥有无限业务权限。

**记录方向**：技术设计记录即可。

### Q36：权限过滤在哪里执行？

**Curie 推荐**：必须在后端查询层强制执行，不能依赖 MCP、CLI 或前端自行过滤。向量检索也必须先限制候选范围或命中后二次校验。

**理由**：入口会变多，权限边界必须集中。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q37：审计日志记录什么？

**Curie 推荐**：记录上传、下载、查询、标签变更、状态变更、归档、手动重试、管理员查看失败资料、token 创建/撤销。字段包括 actor、动作、目标、时间、来源 IP/客户端、结果。

**理由**：资料中枢是可信资料底座，追溯能力比复杂功能更重要。

**记录方向**：写入 `CONTEXT.md`。

### Q38：审计日志能不能被普通管理员修改？

**Curie 推荐**：不能修改，只能追加；如需纠错，追加一条纠错事件。

**理由**：审计日志如果可改，就失去追责价值。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q39：管理员后台第一阶段最小功能是什么？

**Curie 推荐**：资料列表/详情、失败处理查看与重试、归档资料查看、标签清单维护、员工账号启停、员工标签维护、无标签/失败资料队列、审计查询、Skill 目录维护。

**理由**：后台不做分析，只承担纠错、授权、追溯和最小运维。

**记录方向**：写入 `CONTEXT.md`。

### Q40：是否需要普通员工网页前台？

**Curie 推荐**：不需要完整前台。第一阶段普通员工主要通过员工智能体访问；网页只提供必要的上传、确认标签、查看自己上传资料状态。

**理由**：保持智能体优先，避免把资料中枢做成知识库门户。

**记录方向**：写入 `CONTEXT.md`。

### Q41：备份恢复目标怎么定？

**Curie 推荐**：MySQL 每日全量 + binlog/PITR，目标 RPO <= 1 小时，RTO <= 4 小时；对象存储开启版本保留或生命周期保护；向量索引和派生表可从 MySQL + 对象存储重建。

**理由**：元数据不可丢，文件不可误删，派生索引可以牺牲恢复时间换简单性。

**记录方向**：写入“派生索引可重建、对象文件受保护”；建议 ADR。

### Q42：对象存储文件是否允许物理删除？

**Curie 推荐**：第一阶段默认不物理删除；如因合规或误传敏感文件必须删除，应走管理员双确认并留下审计事件。

**理由**：现有共识是归档不删除，但现实中需要保留紧急删除通道。

**记录方向**：写入 `CONTEXT.md`。

### Q43：可观测性第一阶段做到什么程度？

**Curie 推荐**：结构化日志、请求 ID、任务 ID、上传处理耗时、失败率、队列积压、活跃资料数、索引成功率、API p95、鉴权失败次数、对象存储错误率。

**理由**：主要风险在处理链路和权限链路，指标要围绕这两条链路。

**记录方向**：技术设计记录即可。

### Q44：初期容量假设是什么？

**Curie 推荐**：按 20-200 名员工、10-50 个门店/业务标签、10 万级资料目录记录、TB 级对象文件、百万到千万级 chunk 设计；API p95 目标 500ms 内，上传处理允许分钟级完成。

**理由**：这是中小企业资料中枢的合理初始上限，能指导索引和表设计，又不过度工程化。

**记录方向**：写入第一阶段容量假设；建议 ADR。

### Q45：扩展路径是什么？

**Curie 推荐**：先纵向扩 MySQL 和 worker 并发；再拆上传处理队列；再独立向量库/搜索服务；最后才考虑数仓、微服务或多租户。

**理由**：每一步都有明确瓶颈触发，避免一开始背上分布式系统成本。

**记录方向**：技术路线记录即可。

### Q46：第一阶段是否支持多企业/多租户？

**Curie 推荐**：不支持真正 SaaS 多租户；最多预留 `org_id` 字段，但部署和权限模型按单企业使用。

**理由**：多租户会显著增加权限、隔离、备份、审计复杂度。

**记录方向**：写入 `CONTEXT.md`；建议 ADR。

### Q47：最需要补的 ADR 清单是什么？

**Curie 推荐**：
- ADR-001 模块化单体与异步 worker
- ADR-002 资料状态机与 active 可见性
- ADR-003 MySQL/对象存储/派生索引职责划分
- ADR-004 员工账号鉴权与后端强制权限过滤
- ADR-005 上传处理幂等与失败重试
- ADR-006 备份恢复与派生索引可重建

**理由**：这些决策难逆转、没有上下文会显得奇怪，并且都有明确权衡。

**记录方向**：创建 ADR。
